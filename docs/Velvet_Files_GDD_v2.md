# Velvet Files (벨벳 파일즈)
## Merge Detective Agency — Game Design Document

> **장르:** Merge (2-match) + Narrative Mission
> **플랫폼:** HTML5 (모바일 웹 / 세로 화면)
> **문서 목적:** AI 코딩 에이전트(Claude Code 등)가 본 문서만으로 Core Loop이 작동하고 4개 핵심 의뢰가 모두 플레이 가능한 HTML5 프로토타입을 실제 제작할 수 있도록 하는 실행형 기획서.
> **구현 우선순위:** Core Loop 작동 → 4개 의뢰 완전 구현 → 데이터 확장성 → 비주얼 완성도

## 2. 게임 컨셉 및 핵심 재미 가설

### 2.1 세계관 / 톤앤매너
- 1950–60년대풍 빈티지 누아르 무드를 현대적으로 재해석한 도시형 미스터리.
- 플레이어는 매력적인 남성 신사 탐정 **"렌(Ren)"** 의 사무소를 함께 운영하는 파트너 시점.
- 의뢰인은 모두 여성. 가십·로맨스·미스터리·일상 사건을 다루되, **위트와 신사적인 매너**가 톤의 중심.
- 핵심 정서: "이 신사는 어떤 비밀이라도 안전하게 다뤄준다" — 의뢰인이 마음을 여는 안전한 공간감.
- 비주얼 톤: 베이지·로즈우드·딥블루·차콜을 기조로 한 **"젠틀맨 누아르(Gentleman Noir)"**.

### 2.2 핵심 재미 가설 (Core Fun Hypothesis)
> **"내가 만든 단서로 누군가의 사연을 해결한다"는 감정적 몰입 + 머지의 리듬감 있는 손맛.**

- **H1 (호기심):** 의뢰인의 사연이 다음 의뢰를 진행할 동기를 만든다. (스토리 → 머지 진입 동기)
- **H2 (성취):** 머지로 만든 고레벨 아이템을 의뢰인에게 전달할 때 카타르시스가 발생한다.
- **H3 (수집/진행):** 명성(Fame) 상승과 새 의뢰인 등장으로 장기 진행 동기를 형성한다.

### 2.3 차별점
- 일반 머지 게임이 "꾸미기"에 집중하는 반면, 본 게임은 **"미스터리/사연 해결"** 이라는 강한 내러티브 훅을 가짐.
- 의뢰별로 명확한 "사건 종결" 모먼트가 존재 → 짧은 호흡의 성취감 반복.
- 신사 탐정과 여성 의뢰인의 정중한 관계성 → 안전하고 따뜻한 게임 무드.

## 3. Core Loop

### 3.1 한 줄 요약
**의뢰 수락 → 머지로 단서 생성 → 의뢰인에게 전달(포인트 획득) → 사건 종결 → 경험치·명성·코인 → 새 의뢰 등장**

### 3.2 Core Loop 다이어그램

```
┌──────────────────────────────────────────────────────────────┐
│  [아웃게임: 사무소]                                            │
│  의뢰인 입장 → 사연 시청 → 의뢰 수락                           │
│       │                                                       │
│       ▼                                                       │
│  [인게임: 머지 보드 7×9]                                       │
│  생성기 탭 → 아이템 스폰 → 동일 레벨 2개 매치 → 합성           │
│       │                                                       │
│       ▼                                                       │
│  [전달] 버튼 탭 → 자동 매칭 → 수사 포인트 +N                │
│       │                                                       │
│       ▼                                                       │
│  서브 목표 모두 충족? ──No──> 머지 계속                       │
│       │ Yes                                                   │
│       ▼                                                       │
│  [사건 종결 컷] → [결과 화면: XP/Fame/코인 보상]              │
│       │                                                       │
│       ▼                                                       │
│  Fame 등급 변화 시 토스트 + 명패 진화                          │
│       │                                                       │
│       └──────► 다음 의뢰 등장 ────────► (반복)                │
└──────────────────────────────────────────────────────────────┘
```

### 3.3 First-Time User Experience (FTUE)

| Step | 화면 | 액션 | 시간 |
|---|---|---|---|
| 1 | 타이틀 → 사무소 인트로 | 자동 진행 컷신 | 30s |
| 2 | 사무소 (아웃게임) | 첫 의뢰인 "민지" 등장: 잃어버린 고양이 찾기 | 30s |
| 3 | 의뢰 수락 → 머지 보드 진입 | 튜토리얼 핸드 가이드 7단계 (§8.1) | 1m 30s |
| 4 | 머지 보드 | 단서·도구 체인 합성, Requirements 충족 후 [전달] 탭 | 3–4m |
| 5 | 사건 종결 컷 → 결과 화면 | 일러스트 2초 + 보상 (XP 100, 코인 50, Fame +30) | 30s |
| 6 | 사무소 복귀 | 명패 ★1, 다음 의뢰(유리) 등장 알림 | 30s |

**총 FTUE 시간: 약 6–7분.** 이 시간 내에 Core Loop을 1회 완주.

#### 4.2.4 혜진 (Q4)

## 5. 게임 시스템 상세

### 5.1 인게임: 머지 보드

#### 5.1.1 보드 사양
- **그리드:** 7열 × 9행 = 63셀 (세로 모바일에 최적화)
- **셀 크기:** 정사각형, 화면 너비의 1/7 (반응형, 약 100×100px @ 720 width)
- **셀 상태:** `empty` | `item` | `generator` | `client_slot`

#### 5.1.2 아이템 시스템
- 모든 아이템은 **체인(Chain)** 에 속하며, 각 체인은 1–6 레벨을 가짐.
- 동일 체인 + 동일 레벨 아이템 2개를 겹치면 → 다음 레벨 1개로 합성.
- 합성 시 시각/사운드 피드백 발생.
- **보관함:** 보드 외부에 6칸 고정.

**보드 아이템 분류:**

| 분류 | 설명 | 머지 | 의뢰 납품 |
|---|---|---|---|
| 단서 아이템 | Chain A 레벨 1–6, 범죄 현장의 물증과 정황 | ✓ | ✓ |
| 도구 아이템 | Chain B 레벨 1–6, 수사에 사용되는 장비 | ✓ | ✓ |
| 흔적 아이템 | Chain C 레벨 1–6, 현장에 남겨진 물리적 흔적 | ✓ | ✓ |
| 라이프스타일 아이템 | Chain D 레벨 1–6, 의뢰인과 관련된 생활 단서 | ✓ | ✓ |
| 생성기 | 해당 체인 L1 아이템을 보드에 스폰 (§5.1.3 참조) | ✗ | ✗ |
| **제보함 (Tip Box)** | **외부 정보원이 시간차로 단서를 보내는 시간 잠금형 보상 시스템 — §5.3.6 참조** | **✗** | **✗** |

제보함은 다른 머지 아이템과 달리 시간 잠금 상태(LOCKED, READY, OPENING, COMPLETED)를 가진다. 머지 보상으로 보드에 등장하며, 자체 머지는 불가능하다. 자세한 작동은 §5.3.6 참조.

#### 5.1.3 생성기 시스템

**기본 동작:**
- 더블탭 1회 = 인접한 빈 셀에 해당 체인 L1 아이템 1개 스폰.
- 생성기는 자체 체인을 가짐 (예: 단서 생성기 → 단서 체인 L1).
- **자동 생성 주기: 5.0초** (충전 잔량이 1 이상일 때 자동 발동).
- **충전(Charge) 시스템:**
  - 초기 충전: 생성기별 상이 (§6.2)
  - 1회 생성 = 1 Charge 소비
  - 충전 0 → 일정 시간 후 자동 +1 회복
  - **프로토타입 v1은 무한 충전으로 시작**, Phase 6에서 활성화.

**스폰 위치 알고리즘 (4방향 Manhattan + 전체 보드 순회):**

```typescript
function findSpawnCell(generator: Cell, board: Board): Cell | null {
  // 1순위: 4방향 인접 (상·우·하·좌 순)
  const directions: [number, number][] = [
    [ 0, -1], [ 1,  0], [ 0,  1], [-1,  0],  // N, E, S, W
  ];
  for (const [dx, dy] of directions) {
    const c = board.getCell(generator.x + dx, generator.y + dy);
    if (c && c.isEmpty()) { return c; }
  }
  // 2순위: 전체 보드 순회 — 좌→우, 위→아래 순, 맨해튼 거리 최솟값 셀 선택
  return board.findNearestEmptyManhattan(generator.x, generator.y);
}
```

**보드 가득 참 처리:** 빈 셀이 단 한 개도 없으면 "보드가 가득 찼습니다" 토스트 표시 후 탭 무효 처리.

**스폰 시각 피드백:** 해당 셀에 0.2초 화이트 플래시 + 스케일 펄스(1.0 → 1.15 → 1.0).

#### 5.1.4 드래그앤드롭 상호작용 매트릭스

| 출발 ↓ \ 도착 → | 빈 셀 | 같은 체인+같은 레벨 | 같은 체인+다른 레벨 | 다른 체인 아이템 | 생성기 | 의뢰인 슬롯 (가능 체인) | 의뢰인 슬롯 (불가 체인) |
|---|---|---|---|---|---|---|---|
| 보드 아이템 | **이동** | **머지 ⭐** | 위치 교환 | 위치 교환 | 거부 (원위치) | **전달 (포인트 가산)** | 거부 + 토스트 |
| 인벤토리 → 보드 | 이동 (인벤토리 비움) | 머지 (인벤토리 비움) | 위치 교환 | 위치 교환 | 거부 | 전달 | 거부 + 토스트 |
| 보드 → 인벤토리 | 이동 (인벤토리 슬롯에 적재) | (인벤토리 머지 미지원) | 거부 | — | — | — | — |
| 생성기 자체 | (생성기는 드래그 불가) | — | — | — | — | — | — |

> **머지 ⭐ 우선:** 같은 체인+같은 레벨이 정확히 일치할 땐 무조건 머지 우선. 위치 교환 효과는 발생 X.

**시각 피드백 규칙:**

| 상태 | 시각 표현 | 색상 |
|---|---|---|
| 드래그 중 (들고 있는 아이템) | 80% 투명도, 1.1배 스케일, 그림자 | — |
| 드롭 가능 (이동/교환) | 셀 외곽선 2px, 정적 | `#6B8E5A` |
| 머지 가능 ⭐ | 셀 외곽선 3px, 황금 펄스(0.6초 주기) | `#D9B382` |
| 거부됨 | 셀 외곽선 빨강, 0.3초 흔들림 | `#B33A3A` |
| 전달 가능 (의뢰인) | 의뢰인 슬롯 전체 황금 글로우 | `#D9B382` |

**엣지 케이스:**
- 드래그 중 화면 밖 이탈 → 원위치 복귀
- 드래그 중 입력 떼기 (셀 외) → 원위치 복귀
- 드롭 셀이 자기 자신 → 아무 동작 없음
- 멀티터치 → 첫 입력만 유지, 두 번째 무시

#### 5.1.4.1 합성 드롭 자유도 — STRICT vs LENIENT

체인 길이에 따라 드롭 위치 제약이 달라진다:

| 체인 길이 | 모드 | 드롭 가능 위치 |
|---|---|---|
| 2개 | STRICT | 시작 단서의 인접 셀만 |
| 3개 이상 | **LENIENT** | **전체 보드 빈 셀 어디든** |

**메타포:** 단서가 충분히 모이면(3개 이상), 탐정은 *확신에 찬 배치*를 한다. 인접 위치 제약 없이 보드 어디에든 결과를 놓을 수 있다. 짧은 체인(2개)은 여전히 *신중한 배치*를 요구한다.

이 정책은 모바일 환경에서 mistouch를 줄이면서, 사용자가 큰 체인을 시도할 동기를 강화한다.

#### 5.1.4.2 입력 임계값

| 항목 | 값 | 단위 | 설명 |
|---|---|---|---|
| 드래그 시작 거리 | 5 | px | 픽업 후 5px 이상 이동해야 드래그로 인식 |
| 입력 디바운싱 | 50 | ms | 연속 입력 방지 (≈3 프레임 @60fps) |
| 스냅 거리 | 셀 크기의 50% | px | 셀 중심에서 이내면 자동 스냅 |

#### 5.1.4.3 체인 길이별 시각 피드백 (Hold Stage)

체인 길이에 따라 시각 피드백이 단계적으로 강화된다:

| 체인 길이 | Stage | 시각 효과 |
|---|---|---|
| 2 | (없음) | 기본 — 단서 글로우 약함 |
| 3 | 1 | 글로우 살짝 강화 + 미세 진동 |
| 4 | 2 | 글로우 중간 + 진동 강화 |
| 5 | 3 | 글로우 강함 + 화면 살짝 흔들림 |
| 6+ | 4 | 글로우 최대 + 황금 테두리 |

공식: `Stage = clamp(체인 길이 - 2, 1, 4)`

이 단계화는 사용자가 *체인이 자라고 있음*을 직관적으로 인지하게 하고, 큰 체인의 카타르시스를 시각적으로 보상한다.

### 5.2 의뢰 시스템

#### 5.2.1 의뢰 시스템 개요

렌의 사무소로 의뢰인이 찾아온다. 각 의뢰인은 저마다의 사연과 간절함을 안고 문을 두드린다. 렌은 의뢰를 거절하지 않는다. 단지 순서대로, 성실하게, 끝까지 수사를 완수할 뿐이다.

**의뢰인 4인:**

| 의뢰인 | 의뢰 | 톤 |
|---|---|---|
| 민지 | Q1 사라진 미오 | 통통 튀는 활기, 20대 초반 |
| 유리 | Q2 그 사람의 마음 | 조용하고 서정적, 20대 중반 |
| 수아 | Q3 남편의 비밀 | 절제된 단호함, 30대 초반 |
| 혜진 | Q4 사라진 별빛 | 차갑고 우아함, 40대 초반 |

의뢰는 Q1 → Q2 → Q3 → Q4 순서로 순차 진행하며, 동시에 1개만 활성화된다. 거절은 불가능하다. 각 의뢰의 상세 대사·서브목표는 §7을 참조.

**사무소 화면 구성:**
- **렌 일러스트:** 화면 좌측 고정.
- **의뢰 카드:** 화면 우측. 의뢰인 초상 + 이름 + 사연 요약 + 보상 + [수락] 버튼.
- **사건 보드 (Cold Case Wall):** 화면 상단 영역. 해결한 사건의 신문 스크랩 누적 (v2 옵션).
- **명함:** 책상 위 표시. Fame 등급에 따라 외형 진화.
- **HUD:** 상단에 Fame ★등급, 코인 표시.

#### 5.2.2 의뢰 자료구조

```typescript
interface Order {
  id: string;
  type: 'Q1' | 'Q2' | 'Q3' | 'Q4' | 'Special';
  difficulty: '★' | '★★' | '★★★';  // UI 노출용 난이도 등급
  requirements: OrderRequirement[];  // 동시 활성 요구사항 목록
  state: 'Active' | 'Deliverable' | 'Completed';
  rewardCoin: number;
  rewardFame: number;
}

interface OrderRequirement {
  itemChainId: number;
  itemLevel: number;
  requiredCount: number;
  currentCount: number;  // 보드 매칭 시 실시간 갱신
}
```

> **레거시 참고:** 기존 `SubGoal` / `Quest` 인터페이스는 `OrderRequirement` / `Order`로 대체된다. 데이터 파일(`src/data/quests.ts`)의 구조도 동일하게 갱신한다.

#### 5.2.3 요구사항 (Requirements) 진행 — 병렬 동시 활성

모든 Requirements는 의뢰 수락 즉시 **전부 동시 활성화**된다. 사용자는 자유로운 순서로 각 요구사항을 충족할 수 있으며, 의뢰인 슬롯에 모든 Requirements가 동시 표시된다.

**UI 표시 규칙:**

| 상태 | 표시 |
|---|---|
| 미충족 | `{현재}/{필요}` (예: `0/2`) |
| 충족 완료 | 체크마크(✓) + 진한 색 강조 |
| 전체 충족 | [전달] 버튼 활성화 |

**매칭 규칙:**
- 보드에 매칭 아이템이 생성/합성될 때마다 `currentCount` 실시간 갱신
- 복수 Requirement가 동일 체인을 공유할 경우 → 가장 낮은 레벨 Requirement부터 우선 배정
- 자동 배정 처리 (사용자 직접 선택 불가)

**[전달] 버튼 활성 조건:**
모든 Requirements의 `currentCount ≥ requiredCount` 충족 시에만 활성화.

#### 5.2.4 납품 시스템

*드래그 전달* 방식은 폐지되었다. 대신 **버튼 탭 + 시스템 자동 매칭 + §5.3.7 연출 시퀀스**로 대체한다.

*탐정은 증거를 낱낱이 검토한 뒤, 가장 적합한 단서를 직접 골라 의뢰인에게 건넨다.* 사용자가 직접 단서를 드래그하지 않아도, [전달] 버튼 하나로 이 과정이 자동 의식화(儀式化)된다.

**납품 흐름:**

```
1. 모든 Requirements 충족
   └→ [전달] 버튼 활성화 + §5.3.7.1.1 시각 의식 시작
2. 사용자 [전달] 탭
   └→ §5.3.7 4 Phase 연출 자동 실행
3. 연출 완료
   └→ 보상 가산 (Fame + 코인) + 결과 화면 진입
```

**자동 매칭 정책:**
- 시스템이 보드에서 Requirements를 충족하는 아이템을 자동 식별하여 소비
- 동일 조건 아이템이 복수일 때 → 보드 좌상단에서 우하단 방향 우선 선택
- 소비된 아이템의 보드 셀은 즉시 빈 셀로 전환

#### 5.2.5 점수 / 난이도 시스템

**시스템 내부 계산 (사용자 미노출):**

의뢰가 생성될 때 시스템이 내부적으로 점수를 산출한다.

```
점수 = maxItemLevel × 2
     + totalQuantity
     + rarityBonus      // level ≥ 7: +5 / level ≥ 5: +3
     + generatorBonus   // 스페셜 생성기 사용 시: +2
```

- `maxItemLevel`: 의뢰 내 요구 아이템 중 최대 레벨
- `totalQuantity`: 모든 요구 아이템 수량 합계
- `rarityBonus`: level ≥ 7 시 +5, level ≥ 5 시 +3
- `generatorBonus`: 스페셜 생성기(gen_tool, gen_lifestyle, gen_trace) 사용 시 +2

**사용자 노출 (UI):**

사용자에게는 점수 수치 대신 난이도 등급만 표시된다.

| 등급 | 내부 조건 | 의미 |
|---|---|---|
| ★ | 점수 하위 ~30% | 일반 의뢰 |
| ★★ | 점수 중위 30–70% | 도전 의뢰 |
| ★★★ | 점수 상위 70% 컷오프 통과 | 스페셜 의뢰 자격 |

#### 5.2.6 스페셜 의뢰

> 렌의 명성이 자라며, 더 이상 이름을 밝히지 않는 의뢰인들이 사무소를 찾아온다. 이름 없는 의뢰는 *스페셜 의뢰* 로 분류되며, 정해진 4개 의뢰(Q1–Q4)와 별개로 비정기 등장한다.

**활성화 조건:**

- Fame ★3 이상 도달 시 스페셜 의뢰 시스템 활성화
- 의뢰 후보 풀에서 점수 상위 70% 컷오프를 통과한 의뢰 = 스페셜 자격
- 하위 30%는 일반 의뢰 풀로 회귀

**라운드 시스템:**

- 한 스페셜 의뢰 = 1 라운드
- 라운드 완료 조건:

> "사건의 핵심을 뒷받침하는 증거 12점이 모이면 한 라운드가 종결된다."

- 증거 12점: 스페셜 의뢰 내 모든 Requirements의 누적 충족 카운트 합계

**스페셜 의뢰 UI:**
- 의뢰 카드에 ★★★ 배지 표시
- 의뢰인 이름 대신 "익명의 의뢰인" 표시
- 보상 배율 × 1.5 (일반 의뢰 대비)

#### 5.2.7 의뢰 진행 정책

| 항목 | 정책 |
|---|---|
| 의뢰 순서 | **순차 강제** (Q1 → Q2 → Q3 → Q4 고정) |
| 거절 가능 여부 | **거절 불가** ([수락] 버튼만 표시) |
| 동시 진행 의뢰 | **1개만** |
| 보류 / 일시 정지 | **가능** (머지 보드 [←]로 사무소 복귀, 진행도 보존) |
| 재시작 | **자동 이어하기** (게임 재진입 시 마지막 활성 의뢰 자동 로드) |
| 의뢰 실패 | **없음** (시간 제한·실패 조건 없음) |

스페셜 의뢰는 §5.2.6의 별도 정책 적용. Q1–Q4 정규 의뢰 진행 중에는 등장하지 않으며, Q4 클리어 이후 Fame ★3 조건 달성 시 활성화된다.

### 5.3 진행 시스템 (Progression)

#### 5.3.1 XP / 사무소 레벨
- 의뢰 클리어 시 XP 누적, 레벨업.
- 시스템적 진행 데이터 (디버그/내부용).
- 프로토타입 범위: **Lv.1–Lv.4**.

| 레벨 | 누적 XP | 해금 콘텐츠 | 의뢰 |
|---|---|---|---|
| 1 | 0 | 단서(A) + 도구(B) 체인 | Q1: 사라진 미오 |
| 2 | 100 | 라이프스타일(D) 체인 | Q2: 그 사람의 마음 |
| 3 | 300 | (체인 추가 없음) | Q3: 남편의 비밀 |
| 4 | 700 | 흔적(C) 체인 | Q4: 사라진 별빛 (엔딩) |

#### 5.3.2 명성(Fame) 시스템 — 핵심 진행 자원

**컨셉:** "유명해지면 보다 고난이도의 의뢰 등장"을 명시적 자원·시각 표현으로 노출.

- **자원명:** Fame, 단위 ★(별)
- **획득:** 의뢰 클리어 시 XP와 함께 가산
- **소비:** 없음 (진행도 자원)
- **시각 표현:** HUD 우측에 ★등급 + Fame 게이지

**Fame 등급:**

| 등급 | Fame 임계 | 명칭 | 등장 의뢰 | 의뢰인의 첫 멘트 톤 |
|---|---|---|---|---|
| ★1 | 0–99 | 신참 (Rookie) | Q1 | "여기가... 맞나요?" (조심스러움) |
| ★2 | 100–299 | 동네 탐정 (Neighborhood) | + Q2 | "친구가 추천해줘서 왔어요." |
| ★3 | 300–699 | 소문난 탐정 (Renowned) | + Q3 | "꼭 당신이어야 해서 왔어요." |
| ★4 | 700+ | 도시의 탐정 (City Detective) | + Q4 | "은밀히 처리해 주실 분이 필요해요." |

**의뢰별 Fame 보상:**

| 의뢰 | XP | 코인 | Fame ★ |
|---|---|---|---|
| Q1 사라진 미오 | 100 | 50 | +30 |
| Q2 그 사람의 마음 | 200 | 100 | +80 |
| Q3 남편의 비밀 | 400 | 200 | +200 |
| Q4 사라진 별빛 | 700 | 500 | +400 |
| **누적** | 1400 | 850 | **710 → ★4 도달** |

**시각 변화 (Fame 등급별):**
- 사무소 명패: ★1 흑백 → ★2 로즈우드 → ★3 금박 테두리 → ★4 벨벳+엠보싱
- HUD 헤더: "Velvet Files ★N"
- 결과 화면: "당신은 더 유명해졌습니다 (Fame +N)"
- 새 의뢰 등장: "★N 등급 의뢰가 도착했습니다" 토스트

**구현 단순화 노트:** Fame과 XP는 본질적으로 동일 진행도. 구현 시 `fame = xp`로 둬도 무방. 표기·시각 표현만 분리.

### 5.3.5 에너지 시스템

> **컨셉:** 렌의 집중력. 사건에 몰두할 수 있는 정신적 여력을 나타내는 자원.
> 탭 한 번 = 한 번의 추론 소비. 에너지가 소진되면 렌은 잠시 쉬어야 한다.

#### 파라미터

| 파라미터 | 값 | 비고 |
|---|---|---|
| 회복 속도 | 1 / 120초 | 2분에 1 에너지 |
| 자동 회복 상한 | 100 | 이 값까지만 자연 회복 |
| 초기값 | 100 | 게임 최초 시작 시 만충 |
| 보안 상한 | 9,999 | 보상/구매로만 도달 가능한 절대 상한 |

**자동 회복 상한 100의 의미:** 플레이어는 항상 만충 상태로 새 세션을 시작할 수 있다. 100을 초과하는 에너지는 보상 수령 또는 인앱 구매를 통해서만 누적된다. 100→9,999 구간이 유료 버퍼 영역이다.

#### 소비 규칙

- 보드에서 아이템을 **탭(더블 탭)할 때** 에너지 1 소비
- 에너지 0 상태에서 탭 시 → 회복 대기 안내 UI 표시 (세션 종료 유도)
- 의뢰 수락 / 생성기 탭 / UI 탐색은 소비 없음

#### 컨셉 정당화

누아르 탐정물에서 집중력은 한정된 자원이다. 렌은 무한히 추리하지 않는다. 에너지 소진은 "오늘의 수사는 여기까지" 라는 내러티브 신호로 작동하여 의뢰 호흡(5–10분/세션)을 자연스럽게 구조화한다.

### 5.3.6 제보함 시스템

> **컨셉:** 외부 정보원이 시간차로 단서를 보내오는 메타포.
> 렌이 자리를 비운 사이 사무소 우편함에 도착하는 봉투. 열기까지 시간이 걸린다.

#### 상태 머신

```
LOCKED ──(잠금 해제 조건 충족)──▶ READY
READY  ──(수령 탭)──────────────▶ OPENING
OPENING ──(타이머 만료 또는 스킵)─▶ COMPLETED
COMPLETED ──(보상 수령)──────────▶ LOCKED (다음 사이클)
```

| 상태 | 표시 | 플레이어 액션 |
|---|---|---|
| LOCKED | 자물쇠 아이콘 + 잠금 해제 조건 | 없음 |
| READY | "제보 도착!" 알림 배지 | 탭하여 개봉 시작 |
| OPENING | 타이머 카운트다운 + 봉투 애니메이션 | 대기 또는 스킵 |
| COMPLETED | 봉투 열림 연출 + 보상 | 보상 수령 |

#### 스킵 비용

```
스킵 비용(젬) = MAX(1, CEIL(잔여시간(분) × 0.1))
```

- 잔여 10분 → 1젬 / 잔여 60분 → 6젬 / 잔여 120분 → 12젬
- 최소 1젬 보장

#### 보상 구조

| 제보함 등급 | 타이머 | 보상 내용 |
|---|---|---|
| 일반 제보 | 30분 | Chain A 레벨 3 아이템 1개 |
| 중요 제보 | 2시간 | Chain A/B 레벨 4 아이템 1개 + 코인 |
| 특급 제보 | 8시간 | 레벨 5 아이템 1개 + 에너지 30 |

> 보상 아이템 체인은 현재 진행 중인 의뢰의 요구 체인에 가중치를 둔다.

#### 잠금 해제 조건

- 기본 제보함: Fame ★1 달성 시 개방
- 중요 제보함: Fame ★2 달성 시 추가 슬롯 해제
- 특급 제보함: Fame ★3 달성 시 추가 슬롯 해제
- 동시 운용 슬롯: 최대 3개

#### 컨셉 연결

사무소 우편함 UI는 아웃게임(§9.1) 사무소 씬의 일부로 배치된다. 제보함이 도착하면 사무소 씬에 시각 알림(봉투 아이콘 + 배지)이 표시되어 재방문을 유도한다. 누아르의 '미해결 사건이 천천히 풀리는' 클리셰와 결합한 리텐션 메카닉이다.

#### 5.3.6.4 본 게임 컨셉과의 연결

제보함은 단독 시스템이 아니라 본 게임의 두 핵심 요소와 자연스럽게 결합한다.

- **§5.3.4 사건 보드 (Cold Case Wall)와의 시너지:**
  사건 보드가 *"해결된 사건의 누적"* 을 시각화한다면, 제보함은 *"미해결 사건이 천천히 풀리는 과정"* 을 시각화한다. 두 시스템이 함께 *"시간이 흐르며 사건이 진행된다"* 는 누아르의 핵심 정서를 완성한다.

- **§4.1 렌의 캐릭터와의 결합:**
  제보함은 렌이 신뢰하는 *비공식 정보원 네트워크* 의 결과물이다. 렌의 원칙 ("의뢰인의 비밀은 무덤까지")이 정보원에게도 적용되어, 익명의 제보가 안전하게 전달된다. 사용자는 직접 정보원을 만나지 않으나, 시스템적으로 렌의 *"보이지 않는 인맥"* 의 가치를 체감한다.

### 5.3.7 납품 연출 강화

> **컨셉:** 단서를 의뢰인에게 전달하는 행위를 하나의 시각적 의식(ritual)으로 만든다.
> 탐정이 증거를 제출하는 순간의 긴장감과 해소를 4단계 애니메이션으로 구현.

#### 트리거

SubGoal 달성에 필요한 아이템이 보드에 존재하는 상태에서 **[전달] 버튼** 탭 → 납품 연출 시작.

버튼 활성 조건: 현재 SubGoal 요구 아이템이 보드에 1개 이상 존재할 때만 활성화.

#### 5.3.7.1.1 [전달] 버튼 활성화 시각 의식

납품 연출이 시작되기 *전*, [전달] 버튼이 활성화되는 순간에 다음 시각 의식이 발생한다:

| 시각 요소 | 변화 |
|---|---|
| 의뢰인 초상 | 살짝 빛나는 글로우 효과 (베이지 골드 톤, 0.6초 주기 펄스) |
| 의뢰인 표정 | 기본 → 기대 표정으로 변경 |
| [전달] 버튼 | 비활성 회색 → 활성 컬러 (Primary 로즈우드) |
| 보드의 매칭 단서 | 의뢰인 슬롯 방향으로 살짝 진동 (10px 진폭, 0.4초 주기) |

이 의식은 *"탐정이 정중히 단서를 건네기 직전, 의뢰인이 마음을 열고 기다리는 순간"* 의 메타포를 시각화한다. 사용자가 [전달] 버튼을 탭하면 이 의식이 종료되고 §5.3.7.1의 4 Phase 연출 시퀀스로 진입한다.

#### 4단계 연출 시퀀스

| 단계 | 이름 | 지속시간 | 내용 |
|---|---|---|---|
| 1 | Fly | 400ms | 대상 아이템이 의뢰인 슬롯을 향해 InQuad 이징으로 비행 |
| 2 | Absorb | 즉시 | 의뢰인 슬롯에서 흡수 파티클 + 슬롯 펄스 |
| 3 | Flyer | 즉시 | SubGoal 달성 시 "+N pts" 플라이어 텍스트 팝업 |
| 4 | HUD Fly | 600ms | 포인트 수치가 HUD 게이지를 향해 상승 비행 후 병합 |

**총 연출 시간:** ~1,000ms (플레이어가 기다리는 시간은 Fly 400ms + HUD Fly 600ms)

#### 동시성 정책

- 납품 연출 중 보드 탭 입력 **차단** (연출 완료 후 재개)
- 복수 아이템 납품 시(SubGoal 요구량 > 1) → 아이템별 순차 Fly (50ms 간격)
- 연출 도중 다음 SubGoal이 자동 노출 → 연출 완료 후 처리

## 10. 아트 / UX 가이드

> 본 절은 에이전트가 자체 판단해선 안 되는 영역. 명시된 값을 그대로 사용.

### 10.2 타이포그래피
- **제목:** Serif (Playfair Display 또는 Cormorant)
- **본문:** Sans-serif (Inter, Noto Sans KR)
- 한글 우선, 영문 보조

### 10.3 비주얼 톤
- **의뢰인 초상:** 벡터 일러스트 스타일 (그라데이션 절제, 라인 강조)
- **아이템:** 빈티지 오브제 느낌, 아이소메트릭 톤다운
- **체인별 톤 분리:**
  - 단서/도구/흔적: 차콜·세피아·딥블루 톤 (수사 무드)
  - 라이프스타일: 베이지·페일핑크·라벤더 톤 (라이트 무드)
- **프로토타입 v1:** Placeholder 이미지(단색 원형 + 텍스트 라벨)로 시작 후 교체

### 10.4 UX 판단 가이드
- **드래그 우선:** 모든 상호작용은 드래그/탭 두 가지로 한정. 멀티 터치 금지.
- **피드백 즉각성:** 머지 성공 시 50ms 이내 시각/촉각 피드백.
- **모달 최소화:** 의뢰 수락/종결 외에는 풀스크린 모달 사용 금지.
- **방향:** 세로 고정. 가로모드 미지원.
- **글자 크기:** 본문 최소 14sp. 의뢰 사연은 16sp.
- **비활성 상태 표시:** 비해금 의뢰는 자물쇠 아이콘 + 회색조.

### 11.4 핵심 데이터 인터페이스 (TypeScript)

```typescript
// progression.ts
interface PlayerProgress {
  xp: number;
  fame: number;                          // 진행도 자원 (UX 표현용)
  fameTier: 1 | 2 | 3 | 4;
  coins: number;
  level: number;
  activeQuestId: string | null;
  completedQuestIds: string[];
  tutorialFlags: Record<string, boolean>;
}

// chains.ts
interface ChainDef {
  id: string;
  name: string;
  maxLevel: number;
  itemNames: string[];        // index = level - 1
  spriteKeys: string[];
}

// generators.ts
interface GeneratorDef {
  id: string;
  chainId: string;
  initialCharge: number;
  rechargeIntervalSec: number;
  unlockFameTier: 1 | 2 | 3 | 4;
  spawnAlgorithm: '8way_then_bfs';
}

// quests.ts (전체 §5.2.2 참조)

// 머지 판정 의사코드
function tryMerge(a: Item, b: Item): Item | null {
  if (a.chainId !== b.chainId) return null;
  if (a.level !== b.level) return null;
  if (a.level >= getChain(a.chainId).maxLevel) return null;
  return new Item(a.chainId, a.level + 1);
}
```

### 11.5 좌표 / 입력 처리
- 모든 셀은 `(col, row)` 정수 좌표 + 픽셀 좌표 변환 함수 (`grid.ts`).
- 드래그 시작 시 셀 ID를 픽업, 드롭 시 가장 가까운 셀로 스냅.
- 드롭 처리는 §5.1.4 매트릭스 그대로 적용.

### 13.2 의도적 배제 사유
- **에너지 시스템 도입:** 의뢰 호흡(5–10분/세션)과 결합한 집중력 자원으로 작동. 회복 120s/+1, 자동 회복 상한 100, 초기값 100, 보안 상한 9,999의 4파라미터 모델. 100 자동 회복 상한은 사용자가 항상 만충 상태로 시작하되, 보상/구매로만 100 초과 누적이 가능한 부드러운 캡으로 작동한다.
- **거품 셀 배제 유지 (v2 정책 보존):** 머지 본질 검증을 위한 단순화 정책 유지. v2.x 확장 후보로 보류.
- **제보함 도입:** 외부 정보원이 시간차로 단서를 보내오는 메타포의 시간 잠금형 보상 시스템. 누아르 톤의 '미해결 사건이 천천히 풀리는' 클리셰와 결합. §5.3.6에 시스템 명세 신설.
- **부동산 꾸미기 배제:** Project Makeover의 시각 진행 카타르시스는 **사건 보드(Cold Case Wall)** + **명함 진화** 라는 별도 시스템으로 본 게임 컨셉에 맞게 재해석.

### 15.2 리스크 & 가정
- **R1:** 7×9 보드는 일부 모바일 화면에서 비좁을 수 있음 → 동적 셀 크기로 대응.
- **R2:** v1 무한 충전은 코어 루프 호흡을 단조롭게 함 → Phase 6 충전 활성화 필수.
- **R3:** 의뢰 4종으로 4가지 톤 변화를 모두 검증하기엔 표본이 작을 수 있음 → 플레이테스트 시 톤별 만족도 측정.
- **A1:** 사용자는 §8 7단계 튜토리얼로 머지 메커닉을 학습 가능.
- **A2:** 의뢰 4종으로 핵심 가설(H1–H3)을 검증할 수 있음.

### 15.3 글로서리
- **체인(Chain):** 동일 카테고리에서 합성을 통해 레벨업되는 아이템 계열 (A/B/C/D 4종)
- **수사 포인트:** 의뢰 진행도를 측정하는 인게임 자원. 아이템 전달 시 가산.
- **서브 목표 (SubGoal):** 의뢰 안의 단위 요청. 특정 아이템 N개 전달이 조건.
- **의뢰 (Quest):** 1회 단위의 머지 세션 + 내러티브 단위.
- **사무소 (Office):** 아웃게임 메인 허브 화면.
- **명성 (Fame):** "유명세" 진행 자원. ★1–★4 등급으로 시각화.
- **사건 종결 컷 (Closing Cut):** 의뢰 해결 시 노출되는 풀스크린 일러스트 (2초).
- **사건 보드 (Cold Case Wall):** 사무소 벽면, 해결 사건 신문 스크랩 누적 영역.
