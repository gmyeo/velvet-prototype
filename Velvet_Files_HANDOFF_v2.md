### 1.2 Phase별 후속 명령 (Phase 0–2 완료 후)

#### Phase 3 진입 시
```
Phase 0–2가 완료되어 머지가 작동합니다.
이제 Phase 3 (의뢰 시스템 - 인게임)을 진행합니다.

GDD v2 §11.3 Phase 3 체크리스트, §5.2.2 의뢰 데이터 구조,
§7.2 (Q1 사라진 미오), §5.2.3 Requirements 진행 정책,
§5.2.4 납품 방식, §5.3.7 납품 연출을 그대로 구현해주세요.

이번 단계의 목표:
- Q1이 자동 시작
- Requirements가 모두 동시 활성화되어 자유 순서로 충족 가능
- [전달] 버튼 탭 → 자동 매칭 → 포인트 가산
- 100pts 달성 시 "사건 해결!" 메시지 표시
검증 기준은 C2, C5, C6, C10, C12입니다.

핵심 변경사항 (v1 대비):
- 납품: 드래그 전달 아님 → [전달] 버튼 탭 + 자동 매칭 (§5.2.4)
- 진행: 서브 목표 순차 아님 → Requirements 전체 동시 활성 (§5.2.3)

생성기는 gen_clue + gen_tool 두 개로 확장하고,
의뢰인 슬롯 + Requirements 동시 활성 + [전달] 버튼 + 포인트 게이지를 추가합니다.
```

```
Phase 3가 완료되어 Q1 인게임이 작동합니다.

§9.3 결과 화면 와이어프레임, §5.3.4 사건 종결 컷

씬 전환: Office → Board → ClosingCut → Result → Office
```

## 4. 자산 (Asset) 처리 정책

프로토타입 단계에서 외부 일러스트/사운드 자산은 **모두 placeholder로 시작**합니다.

### 4.1 Placeholder 규칙
| 자산 타입 | Placeholder 형태 |
|---|---|
| 아이템 스프라이트 | 단색 원형 + 체인 컬러 + 레벨 숫자 텍스트 |
| 생성기 스프라이트 | 단색 사각형 + 이모지 (📋 단서, 🔧 도구, ✨ 라이프스타일, 🔬 흔적) |
| 의뢰인 초상 | 컬러 원형 + 이름 텍스트 (민지/유리/수아/혜진) |
| 렌 일러스트 | 회색 실루엣 + "R" 이니셜 |
| 사건 종결 컷 | 단색 배경 패널 + 캡션 텍스트 (GDD §5.3.4 표 그대로) |
| 제보함 | 단색 사각형 + 잠금/열림 아이콘 + 잔여 시간 텍스트 |
| 사운드 | Phase 6까지 모두 무음 |

### 4.2 자산 교체 우선순위 (Phase 6 이후, 운영 단계)
1. 의뢰인 초상 4종 (가장 임팩트 큼)
2. 렌 일러스트 (정면 흉상 1장)
3. 사건 종결 컷 4종
4. 아이템 스프라이트 4 체인 × 6 레벨 = 24종
5. 사운드 (머지 / 스폰 / 종결 / UI)

## 5. 자주 발생할 수 있는 이슈 & 대응

### 5.1 컨텍스트 손실 방지
- Claude Code 세션이 길어지면 GDD 일부를 잊을 수 있음
- 새 작업 시작 시 항상 **"GDD v2 §X.Y에 따라"** 라는 명시적 참조를 포함
- 구현 모호 시 Claude Code가 먼저 GDD 해당 섹션을 다시 인용하도록 유도

### 5.2 GDD와 다른 결정이 필요할 때
- **원칙:** GDD v2 우선. 변경이 필요하면 GDD를 먼저 수정하고 코드는 그 다음.
- 코드만 변경하면 다음 세션에서 다시 충돌 발생.

### 5.3 Phase 건너뛰기 금지
- Phase 0–6은 점진적 빌드. 건너뛰면 검증이 꼬임.
- 각 Phase 종료 시점에 "플레이 가능한 빌드"가 나와야 함.
- Phase X가 끝나면 GDD §14.1의 해당 C 항목 체크 후 다음 Phase 진입.

### 5.4 데이터 ↔ 코드 분리 원칙
- 의뢰 내용 / 체인 / 튜토리얼 등 데이터는 모두 `src/data/*.ts` 파일에만 존재
- 시스템 코드는 데이터를 참조만 할 뿐 하드코딩 금지
- 의뢰 추가/수정 시 코드 수정 없이 데이터 파일만 변경되어야 함

### 5.5 v2 신규 시스템 구현 순서
- 에너지 시스템(§5.3.5): Phase 5에서 활성화 — Phase 1–4에서는 무한 충전 유지
- 제보함(§5.3.6): Phase 5에서 도입 — 머지 보상으로만 보드에 등장
- [전달] 버튼(§5.2.4): Phase 3부터 적용 — 드래그 전달 방식 제거
- LENIENT 드롭(§5.1.4.1): Phase 1–2에서 기초 구현, Phase 3에서 검증
